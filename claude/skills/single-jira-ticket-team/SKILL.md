# Single-Jira-Ticket Team Workflow

A reusable team setup for shipping **one** Jira ticket end-to-end while the user keeps working in the main session: a single implementer in its own git worktree off the base branch, plus a shared reviewer and a shared tester, plus a recurring cron that polls the resulting PR for review feedback.

This is the single-ticket counterpart to `parallel-jira-tickets`. The value isn't parallelism — it's:
- **Backgrounding**: the user can keep using the main session while the team works.
- **Quality gate**: implementer must get APPROVED from reviewer and PASS from tester before committing — catches scope creep, convention drift, and breakage that an inline single-pass implementation often misses.
- **Isolation**: the worktree means the user's main repo isn't touched.

## When to use

- The user gives you **one** Jira ticket / Jira URL / bug description and explicitly wants it handled in the background, or describes scope big enough that they don't want to babysit it.
- The repo has a sensible default branch (`prod`, `main`, `master`) and a way to run tests.

**When not to use:**
- One-line fixes, doc tweaks, or anything that's faster to do inline than to spool up a team. Just edit the file.
- Tickets where the user wants to drive the design themselves — the implementer will make autonomous decisions inside the worktree.
- Ambiguous scope. If the Jira ticket isn't clear, clarify with the user *before* spawning the team — a flailing background agent burns tokens and produces a bad PR.

For ≥ 2 Jira tickets, use `parallel-jira-tickets` instead.

## Required tools

This skill uses tools that are *deferred* in many sessions. **Load them up front** with one ToolSearch call:

```
ToolSearch query: "select:TeamCreate,Agent,SendMessage,TaskCreate,TaskUpdate,TaskList,CronCreate,AskUserQuestion"
```

If the session also needs Jira CLI tools (e.g. `jira issue view`) loaded, add those to the same search.

## Step 1 — Gather inputs

Auto-detect from the project where you can; only ask via `AskUserQuestion` when detection fails or is ambiguous.

| Input | How to detect | Fallback ask |
|---|---|---|
| **Jira ticket** | Provided by the user (e.g. `JIRA-123`) | Required — refuse if missing |
| **Ticket details** | Fetch via `jira issue view <KEY> --plain`. Read enough to know actual scope. | Required |
| **Repo root** | `git rev-parse --show-toplevel` from current cwd | Ask which repo if you're outside one |
| **Base branch** | Read project CLAUDE.md ("Main branch:", "Default branch:"); else `git symbolic-ref refs/remotes/origin/HEAD`; else `prod`/`main` heuristic | Ask if ambiguous |
| **Worktree root** | Project CLAUDE.md memory or convention; the user's worktree-collection directory if one exists | Ask via `AskUserQuestion` with detected siblings as options |
| **Branch prefix** | Read project CLAUDE.md ("Feature branches: `username/...`"); else recent branches via `git for-each-ref --format='%(refname:short)' refs/heads/` | Ask if no clear pattern |
| **Build/test commands** | Project CLAUDE.md "Common Commands" section; else detect (`./gradlew test`, `cargo test`, `pytest`, `npm test`, etc.) | Ask the implementer prompt to figure it out per-module |
| **PR template** | `.github/pull_request_template.md` | None if absent — agent writes a sensible default |
| **AI footer text** | Project CLAUDE.md ("🤖 Generated by …" instruction) | None if absent — agent skips the footer |

**Read the project CLAUDE.md once and pass relevant excerpts into the implementer's prompt** so it doesn't re-discover conventions.

## Step 2 — Verify preconditions

Refuse to proceed if any of:

1. The source clone has uncommitted changes that would be picked up by worktree creation. (`git status -s` non-empty in critical files; ignore obvious untracked-but-fine cases.)
2. The worktree root doesn't exist or isn't writable.
3. The Jira ticket has no clear actionable scope (description is empty / too vague). Better to ask the user to clarify than to spawn an implementer that flounders. **This matters more than in `parallel-jira-tickets`** — there's no other ticket to compensate if this one stalls.

## Step 3 — Create the worktree

```bash
cd <repo-root>
git fetch origin <base-branch>
git worktree add -b <branch-prefix>/<TICKET>-<kebab-description> <worktree-root>/<repo-name>-<TICKET> origin/<base-branch>
```

**Important caveat:** `git worktree add -b … origin/<base>` auto-tracks `origin/<base>`, so the implementer's first push must use `-u origin <branch>` to override the upstream. Bake this into the implementer prompt.

The branch description should come from the ticket title in kebab-case, max ~60 chars total branch length.

## Step 4 — Create the team

```
TeamCreate {
  team_name: "ticket-<TICKET-KEY-lowercased>",
  description: "Implements ticket <TICKET> end-to-end"
}
```

Then create one task with TaskCreate. The task description should include: worktree path, branch name, ticket key/URL, and a brief summary of what to do.

## Step 5 — Spawn agents (all in parallel, all `run_in_background: true`)

Spawn in **one message** with three Agent tool calls. All three must be backgrounded — that's what lets the user keep working in the main session.

### Reviewer (1, shared)
- `subagent_type: general-purpose`
- `name: reviewer`
- Prompt: read project CLAUDE.md once, then idle. On request, read `git diff origin/<base>...HEAD` from the named worktree, review for correctness vs. the ticket and project conventions (no fully-qualified class names, idiomatic Java/Go/etc), reply via SendMessage with `APPROVED — <one line>` or `CHANGES REQUESTED — <numbered list with file:line refs>`. Never edits code.

### Tester (1, shared)
- `subagent_type: general-purpose`
- `name: tester`
- Prompt: idle, then on request `cd` into the named worktree and run the gradle/cargo/pytest/etc command the implementer provides. Reply `PASS — <summary>` or `FAIL — <key excerpt>`. Don't invent tests if the implementer didn't propose any. Doesn't edit code.

### Implementer (1)
- `subagent_type: general-purpose` (custom subagent types can't edit files)
- `name: impl-<ticket-id>`
- Prompt **must include** these strict workflow steps in this order:
  1. `cd` into the assigned worktree.
  2. Read project CLAUDE.md for conventions.
  3. **Investigate** the relevant code before editing — find the actual problem, not just the line in the ticket.
  4. **Implement** the minimal fix. For Java/Gradle projects, run `./gradlew spotlessApply`. For other languages, run the project's formatter (`gofmt`, `cargo fmt`, `prettier`, etc.).
  5. **Plan tests** — identify or write a test that covers the fix. Don't claim "tested" if you didn't actually exercise the change.
  6. **Code review loop:** SendMessage `reviewer` with worktree path + 2-3 line summary of the change. Loop until APPROVED.
  7. **Test loop:** SendMessage `tester` with worktree path + the exact test command. Loop until PASS.
  8. **Commit + push + draft PR** (only after both APPROVED + PASS):
     - Stage specific files, never `git add -A`.
     - Conventional Commits (`fix(scope): …`, `refactor(scope): …`, etc.) plus the project's AI co-author footer if specified in CLAUDE.md.
     - Never `--no-gpg-sign`, never `--no-verify`, never `--amend`, never force-push.
     - First push: `git push -u origin <branch>` (overrides the auto-tracking from worktree-add).
     - `gh pr create --draft` with title `[<TICKET>]: <short description>` (or whatever convention the project uses), filling the PR template.
     - If the project asks for it, post `@codex review` (or equivalent) as a PR comment after creation — comments trigger Codex on draft PRs; descriptions don't.
  9. **Report** the PR URL back to `team-lead` via SendMessage and mark the task complete via TaskUpdate.
- Pause-and-ask rules: if scope balloons, the fix is much larger than expected, or the migration touches infra they didn't anticipate, **pause and SendMessage `team-lead`** before going further. Don't ship a half-fix; don't expand scope silently. **For single-ticket workflows the user is by default not watching** — when in doubt, pause and ask rather than guessing.

### Common across all agents
- Refer to teammates by name, never UUID.
- Stay idle between turns — don't poll, don't busy-loop.
- Don't add comments explaining what code does. Don't add backwards-compat shims. Don't write planning/decision/analysis markdown files.

## Step 6 — Set up the PR-comment polling cron

After the PR is open, register a recurring cron to re-dispatch the implementer to poll the PR for new review comments. **Classify by author** and route differently — don't act on human comments without the user's say-so:

```
CronCreate {
  cron: "<off-minute> * * * *",   // e.g. "23 * * * *" — avoid :00 and :30
  recurring: true,
  durable: false,                  // session-bound; the team is too
  prompt: "Hourly PR-comment poll for team `<team-name>`. SendMessage the implementer with these per-comment classification rules:\n\n1. **Codex / AI reviewer** (`user.login == \"codex\"` or starts with `chatgpt`): AUTO-HANDLE. Fix as a new commit on the same branch (Conventional Commits, project AI co-author footer if specified). Reply on the thread with the AI footer pointing at the new SHA. Don't resolve the thread.\n2. **Human teammate** (any non-bot, non-AI login): SURFACE, don't act. SendMessage `team-lead` with author login, file:line / thread URL, the comment quoted verbatim, and a one-line interpretation. Don't auto-fix. Don't auto-reply on the user's behalf.\n3. **Bots** (devflow, github-actions, dependabot, codeowners-auto, integration bots, plain approvals/LGTM with no actionable content): skip silently.\n\nReport back: how many Codex items auto-fixed (with new SHAs), how many human items surfaced, how many skipped. If the implementer is no longer alive (new session, team gone), report that to the user — don't try to respawn."
}
```

Pick an off-minute (`:17`, `:23`, `:47`, etc.) — avoid `:00` and `:30` to dodge the global fleet stampede. Tell the user about the **7-day auto-expiry** on recurring crons and that the cron is **session-bound** (dies when this Claude session exits).

The split between AI and human authors is deliberate: Codex feedback is mechanical and worth auto-fixing on the spot; human teammate feedback often carries context, suggestions, or questions that the user wants to read and reply to personally. If the user asks for the agent to address human feedback automatically too, swap rule 2 to AUTO-HANDLE — but default is SURFACE.

## Step 7 — Hand off and go idle

Tell the user, briefly:
- The team name and the worktree path.
- The PR URL once it's open (it'll arrive as an implementer message).
- The cron job ID (so they can `CronDelete` it).
- That further `@codex review` requests, additional comments to address, or scope-blocker pings will be relayed as teammate messages.

Then **go idle**. Do not poll the team yourself — messages from teammates arrive automatically as new conversation turns. The user can continue working in the main session; the implementer's progress, the PR URL, and review-comment summaries will surface as inbound messages without you having to ask.

## Anti-patterns

- **Don't** use this skill for trivial work — just edit the file inline. The team setup overhead exceeds the benefit for one-line fixes.
- **Don't** spawn the implementer foreground. The whole point is backgrounding — `run_in_background: true` is mandatory on all three agents.
- **Don't** spawn the implementer without a reviewer and tester — the gate-loop is what makes this pattern reliable.
- **Don't** use `EnterWorktree` — that puts the worktree inside the source repo. Use `git worktree add` to a separate folder.
- **Don't** skip the `-u origin <branch>` on first push — `worktree add -b … origin/<base>` makes the branch track the base, and the first push will go to the wrong place.
- **Don't** forget the deferred-tool load. `TeamCreate` and friends won't be callable without the ToolSearch step.
- **Don't** auto-resolve PR review threads on the user's behalf. Reply, fix, push — let the user resolve.
- **Don't** auto-fix or auto-reply to human-teammate review comments by default — surface them to the user. Codex / AI-reviewer comments are the exception; those are mechanical enough to handle on the spot.
- **Don't** silently expand scope when the ticket turns out to be bigger than expected. Pause and message `team-lead`. With a single-ticket team the user isn't watching by default — guessing wrong wastes the whole session.

## Variant: language-specific overrides

When the project is **clearly** of a known type, swap a few defaults:

| Project | Formatter | Test runner | Lint check |
|---|---|---|---|
| Gradle/Java | `./gradlew spotlessApply` | `./gradlew :module:test --tests "Class.method"` | `./gradlew spotlessCheck` |
| Go | `go fmt ./...` | `go test ./module/...` | `go vet ./...` |
| Rust | `cargo fmt` | `cargo test -p crate test_name` | `cargo clippy` |
| Node.js | `npx prettier --write .` | `npm test -- --testNamePattern` | `npm run lint` |
| Python | `ruff format .` (or `black .`) | `pytest path/to/test.py::test_name` | `ruff check` |

If multiple build systems coexist (e.g. Gradle + Bazel in the same repo), default to whichever the project CLAUDE.md highlights, and let the implementer escalate if the wrong one was picked.
