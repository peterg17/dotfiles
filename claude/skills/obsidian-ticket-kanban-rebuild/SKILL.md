# Rebuild Obsidian Ticket Kanban

Claude counterpart to pi's `obsidian_ticket_kanban_rebuild` tool. Reads all `type: ticket` notes from the configured scan dirs and regenerates the mobile-friendly Kanban board at `$OBSIDIAN_TICKETS_KANBAN` (default `00 Maps/Agentic Tasks Kanban.md`). Does not migrate or modify ticket notes themselves; pair with `obsidian-ticket-rebuild` for that.

Pi's `pi/extensions/obsidian-tickets/index.ts` is the canonical source for the Kanban template, column ordering, priority badges, and done-limit logic. Update both sides in lockstep if you change either.

## When to use

- User wants the Kanban refreshed after creating/updating tickets from Claude.
- The Agentic Tasks dashboard is fine but the Kanban view (mobile-optimized) is stale.
- User says "open the Kanban on my phone after this update" — refresh before they look.

**When not to use:**
- User wants the full dashboard and Kanban refreshed together — use `obsidian-ticket-rebuild` (which optionally rebuilds the Kanban when it exists).
- The Kanban file does not exist and `$OBSIDIAN_TICKETS_KANBAN` is unset — confirm with the user whether to create it first. Pi treats "create the Kanban for the first time" as an explicit one-time action.

## Required tools

`Bash`, `Read`, `Write`. No deferred tool loading required.

## Step 1 — Resolve vault config

Same resolution as `obsidian-ticket-list`:

- vault root: `$OBSIDIAN_TICKETS_VAULT` → `$OBSIDIAN_VAULT_ROOT` → `~/Documents/notes`.
- kanban path: `$OBSIDIAN_TICKETS_KANBAN` → `00 Maps/Agentic Tasks Kanban.md`.
- ticket dir: `$OBSIDIAN_TICKETS_DIR` → `01 Projects/Tickets`.
- scan dirs: `$OBSIDIAN_TICKETS_SCAN_DIRS` → `[ticket dir]`, plus `ticket-scan-dirs:` from `$OBSIDIAN_TICKETS_DASHBOARD` frontmatter.
- done limit: `$OBSIDIAN_TICKETS_KANBAN_DONE_LIMIT` → `20`. Must parse as a positive integer; else use `20`.

## Step 2 — Collect tickets

Walk scan dirs, read each `*.md` file, keep only those with `type: ticket`. Build the canonical `TicketMeta` for each (same logic as the list/rebuild skills). Do **not** mutate the source files.

## Step 3 — Render the Kanban

Write to `<vault>/<kanban path>`. The exact layout (matching pi's `renderKanbanBoard`):

### Kanban frontmatter

```yaml
---
kanban-plugin: board
type: dashboard
dashboard: agentic-tasks-kanban
updated: <today YYYY-MM-DD>
tags:
  - agentic/tasks
  - agentic/kanban
ticket-scan-dirs:
  - <scan dir 1>
  - <scan dir 2>
---
```

### Body header

```md
%% Generated by the obsidian-tickets Pi extension. Ticket notes are the source of truth; manual card moves are overwritten on rebuild. %%
%% Done and archived columns show the <N> most recently updated tickets for mobile readability. Rebuild with /tickets-kanban-rebuild. %%
```

`<N>` is the resolved done limit.

### Columns

Emit one `## <Column Heading>` section per status, in this order: `in-progress`, `needs-review`, `blocked`, `todo`, `done`, `archived`, then any unknown statuses alphabetically. Always include every status from the canonical list even if it's empty (matches pi).

Column heading is the human label:

- `in-progress` → `In Progress`
- `needs-review` → `Needs Review`
- `blocked` → `Blocked`
- `todo` → `Todo`
- `done` → `Done`
- `archived` → `Archived`
- unknown → Title-Case the status (replace `-` with space, capitalize each word)

### Cards

Within each column, filter tickets to that status and sort by:

1. Priority order: `urgent`, `high`, `medium`, `low`, others last (priority sort key).
2. Then `updated` descending.
3. Then title ascending.

For `done` and `archived` columns, override the sort to:

1. `updated` descending.
2. Title ascending.

Then truncate to the done limit (default 20). If the original count exceeds the truncated count, prepend a one-line HTML comment **before** the cards (still inside the column section):

```html
<!-- Showing <shown> of <total>; older <status> tickets are hidden for mobile readability. -->
```

For each card, emit a checklist item:

```
- [<checkbox>] [[<path-without-.md>|<title>]]<br><sub><priority badge> <priority> · <project> · updated <updated>[ · PR <pr>]</sub>
```

- `<checkbox>` is `x` if status is `done` or `archived`, else a single space.
- Priority badge: `urgent`=`🔴`, `high`=`🟠`, `medium`=`🟡`, `low`=`🟢`, anything else=`⚪️`.
- The `<sub>...</sub>` block uses HTML-escaped values: `&` → `&amp;`, `<` → `&lt;`, `>` → `&gt;`, `"` → `&quot;`.
- Join the metadata pieces with ` · `. Include `PR <pr>` only if `pr` is non-empty.
- The wikilink uses path-without-`.md`. Always include the `|<title>` alias even if equal to the basename — pi does for consistency.

If a column has zero tickets (after truncation), emit:

```
%% No tickets in this status. %%
```

### Settings footer

At the very end of the file:

```
%% kanban:settings
```
```
{
  "kanban-plugin": "board",
  "list-collapse": []
}
```
```
%%
```

Important: the JSON block is fenced with **plain triple-backticks (no language tag)**. Adding a language like ```json` makes the mobile Kanban parser treat the word "json" as JSON content (this exact bug is the reason pi avoids the language tag — see comment in `renderKanbanSettings`).

## Step 4 — Write and report

Write the file atomically via `Write`. Refuse if the resolved kanban path is a symlink (matches pi's `safeVaultWritePath` behavior).

Report:

- the kanban path that was (re)written,
- `tickets rendered: <N>`,
- a one-line note that ticket notes remain the source of truth and that any manual Kanban card moves get overwritten on next rebuild.

## Anti-patterns

- **Don't** write outside the vault root. Refuse symlinks.
- **Don't** use a language tag in the kanban:settings fenced code block. This is a known mobile-parser bug; pi explicitly avoids it.
- **Don't** modify ticket notes here. This skill only writes the Kanban file.
- **Don't** drop or reorder unknown statuses — append them alphabetically after the canonical 6.
- **Don't** truncate `in-progress`, `needs-review`, `blocked`, or `todo` columns. The done-limit only applies to `done` and `archived`.

## Safety rules

- Vault root resolution outside the user's home directory requires confirmation.
- Treat ticket bodies and links as sensitive — only report the kanban path and ticket count.
- Pi's `pi/extensions/obsidian-tickets/index.ts` functions `renderKanbanBoard`, `renderKanbanSettings`, `kanbanCard`, `humanizeStatus`, `escapeHtml`, `kanbanColumns`, `sortTicketsForKanban` are the canonical reference. Mirror any pi-side change here.
