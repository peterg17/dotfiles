# Obsidian Ticket Team

Claude counterpart to the pi `obsidian-ticket-team` skill. Coordinates implementation work described by Obsidian Markdown ticket notes using Claude-native primitives (`TeamCreate`, `Agent`, `SendMessage`, `CronCreate`).

This is the Obsidian-sourced sibling of `single-jira-ticket-team` / `parallel-jira-tickets`: the team setup is the same, but the ticket lives as a Markdown note in a local vault rather than in Jira.

## When to use

- The user gives you an Obsidian note path, a `[[wikilink]]`, a vault-relative path, or pasted ticket Markdown and wants the work shipped.
- The note describes implementation work that needs code changes plus a PR.
- The repo target is identifiable from the note (frontmatter `repo:`), context, the current working directory, or a quick clarification.

**When not to use:**
- The note is a planning/brainstorming doc, not a ticket. Just discuss inline.
- The work is a one-line fix or doc tweak — faster to edit the file than to spin up a team.
- There's no PR target (no repo, or the work is purely note-editing). Use ordinary file tools instead.

For Jira-tracked tickets use `single-jira-ticket-team` or `parallel-jira-tickets` instead. The Obsidian variant is for vault-sourced work where the note itself is the source of truth.

## Required tools

This skill uses tools that are *deferred* in many sessions. **Load them up front** with one ToolSearch call:

```
ToolSearch query: "select:TeamCreate,Agent,SendMessage,TaskCreate,TaskUpdate,TaskList,CronCreate,AskUserQuestion"
```

Unlike the pi version there are no `obsidian_ticket_*` tools in Claude — note reads/writes go through ordinary `Read` and `Edit` tools.

## Step 1 — Resolve the ticket note

The user may provide:

- an absolute Markdown path,
- a path relative to the configured vault,
- a wiki link such as `[[Ticket Title]]`,
- a Markdown link to a local note,
- or pasted ticket Markdown.

Resolution order:

1. If an absolute path is provided and the file exists, read it.
2. For wiki links or relative paths, search likely vault roots: `$OBSIDIAN_TICKETS_VAULT`, `$OBSIDIAN_VAULT_ROOT`, the current working directory, and any path the user mentions.
3. If multiple notes match, ask the user to choose via `AskUserQuestion`.
4. If only pasted Markdown is available, proceed from that content but **ask before writing updates anywhere** — there's no canonical note file to update.

## Step 2 — Gather inputs from the note

Read the note and extract:

| Input | How to detect | Fallback ask |
|---|---|---|
| **Note path(s)** | Resolved in Step 1 | Required — refuse if missing |
| **Title** | First `# ` heading or filename | Use filename |
| **Problem / acceptance criteria** | `## Problem`, `## Acceptance Criteria` sections | Ask the user if absent or empty |
| **Target repo** | Frontmatter `repo:` field; else current cwd if it's a git repo *and matches the note*; else ask | Required — refuse if no repo can be determined |
| **Base branch** | Frontmatter `branch:` parent; else project CLAUDE.md ("Main branch:"); else `git symbolic-ref refs/remotes/origin/HEAD`; else `prod`/`main` heuristic | Ask if ambiguous |
| **Feature branch** | Frontmatter `branch:` field; else derive a short kebab-case branch from the note title | None — generate |
| **Worktree root** | Project CLAUDE.md memory or convention; sibling-of-repo directory the user has used before | Ask via `AskUserQuestion` |
| **Build/test commands** | Project CLAUDE.md "Common Commands"; else detect (`./gradlew test`, `cargo test`, `pytest`, `npm test`, etc.) | Ask the implementer prompt to figure it out |
| **PR template** | `.github/pull_request_template.md` | None if absent — agent writes a sensible default |
| **AI footer text** | Project CLAUDE.md ("🤖 Generated by …" instruction) | None if absent — agent skips the footer |

**Read the project CLAUDE.md once and pass relevant excerpts into the implementer's prompt** so it doesn't re-discover conventions.

## Step 3 — Verify preconditions

Refuse to proceed if any of:

1. The note has no actionable scope (problem/acceptance empty or too vague). Ask the user to clarify the note first.
2. The target repo can't be determined and the user can't (or won't) clarify.
3. The source clone has uncommitted changes that would be picked up by worktree creation. (`git status -s` non-empty in critical files; ignore obvious untracked-but-fine cases.)
4. The worktree root doesn't exist or isn't writable.
5. More than ~5 notes were requested in one batch. Ask the user to batch the rest separately.

## Step 4 — Prepare the note (minimal additive updates)

Before spawning agents, make minimal additive updates with `Edit`:

- set `status: in-progress` in the frontmatter,
- ensure `type: ticket`, `## Work Log`, and `## PR` sections exist (add them if missing),
- append a work-log entry that a Claude team was started.

Do **not** rewrite the user's problem statement or unrelated note content. The team lead (you) is the only writer of note updates — implementers don't touch the note.

If the user provided only pasted Markdown, skip Step 4 and ask whether/where to persist the note before proceeding.

## Step 5 — Create the worktree(s)

For each note, fetch the latest base branch then add a worktree:

```bash
cd <repo-root>
git fetch origin <base-branch>
git worktree add -b <feature-branch> <worktree-root>/<repo-name>-<short-slug> origin/<base-branch>
```

**Important caveat:** `git worktree add -b … origin/<base>` auto-tracks `origin/<base>`, so the implementer's first push must use `-u origin <branch>` to override the upstream. Bake this into the implementer prompt.

The short slug should come from the note title in kebab-case, max ~60 chars total branch length.

## Step 6 — Create the team

```
TeamCreate {
  team_name: "obsidian-<short-slug>",
  description: "Implements Obsidian ticket(s) <list>"
}
```

Then create one task per note with TaskCreate. Each task description should include: worktree path, branch name, note path, and a brief summary of what to do.

## Step 7 — Spawn agents (all in parallel, all `run_in_background: true`)

Spawn in **one message** with multiple Agent tool calls.

### Reviewer (1, shared)
- `subagent_type: general-purpose`
- `name: reviewer`
- Prompt: read project CLAUDE.md once, then idle. On request, read `git diff origin/<base>...HEAD` from the named worktree, review for correctness against the **note's problem and acceptance criteria** (not just generic code review), reply via SendMessage with `APPROVED — <one line>` or `CHANGES REQUESTED — <numbered list with file:line refs>`. Never edits code or the note.

### Tester (1, shared)
- `subagent_type: general-purpose`
- `name: tester`
- Prompt: idle, then on request `cd` into the named worktree and run the exact command the implementer provides. Reply `PASS — <summary>` or `FAIL — <key excerpt>`. Don't invent tests if the implementer didn't propose any. Doesn't edit code or the note.

### Implementer(s) — one per note
- `subagent_type: general-purpose` (custom subagent types can't edit files)
- `name: impl-<short-slug>`
- Prompt **must include** these strict workflow steps in this order:
  1. `cd` into the assigned worktree.
  2. Read project CLAUDE.md and any project-local `AGENTS.md` for conventions.
  3. **Investigate** the relevant code before editing — find the actual problem from the note's `## Problem` section, not just the surface description.
  4. **Implement** the smallest correct change that satisfies the acceptance criteria. Run the project's formatter (`./gradlew spotlessApply`, `gofmt`, `cargo fmt`, `prettier`, `ruff format`, etc.).
  5. **Plan tests** — identify or write a test that exercises the acceptance criteria. Don't claim "tested" if you didn't actually run anything.
  6. **Code review loop:** SendMessage `reviewer` with worktree path + 2-3 line summary + a quoted excerpt of the acceptance criteria. Loop until APPROVED.
  7. **Test loop:** SendMessage `tester` with worktree path + the exact test command. Loop until PASS.
  8. **Commit + push + draft PR** (only after both APPROVED + PASS):
     - Stage specific files, never `git add -A`.
     - Conventional Commits (`fix(scope): …`, `feat(scope): …`, etc.) plus the project's AI co-author footer if specified in CLAUDE.md.
     - Never `--no-gpg-sign`, never `--no-verify`, never `--amend`, never force-push.
     - First push: `git push -u origin <branch>` (overrides the auto-tracking from worktree-add).
     - `gh pr create --draft` with title derived from the note title, filling the PR template; include the note path or wikilink in the PR description so reviewers can trace context.
     - If the project asks for it, post `@codex review` as a PR comment after creation — comments trigger Codex on draft PRs; descriptions don't.
  9. **Report** to `team-lead` via SendMessage: changed files, validation run, commit SHA, PR URL. Mark the task complete via TaskUpdate.
  10. **Do not edit the note.** The team lead handles all note updates to avoid concurrent rewrites.
- Pause-and-ask rules: if scope balloons, the note's acceptance criteria turn out to be ambiguous, or the work touches infra the note didn't anticipate, **pause and SendMessage `team-lead`** before going further.

### Common across all agents
- Refer to teammates by name, never UUID.
- Stay idle between turns — don't poll, don't busy-loop.
- Don't add comments explaining what code does. Don't add backwards-compat shims. Don't write planning/decision/analysis markdown files.
- **Don't edit the Obsidian note.** Only the team lead writes note updates.

## Step 8 — Track progress in the note

As implementer messages arrive, append concise work-log entries via `Edit`:

```md
- 2026-05-13 14:22 — Status: in-progress. Spawned team `obsidian-fix-foo`.
- 2026-05-13 14:41 — Review: APPROVED by reviewer.
- 2026-05-13 14:53 — Tests: PASS `./gradlew :module:test`.
- 2026-05-13 14:55 — PR: https://github.com/org/repo/pull/123
```

When a PR opens:

- set/update the `pr:` frontmatter field with the URL,
- add the PR URL under `## PR`.

When complete:

- check off acceptance criteria *only* if they're actually satisfied,
- set `status: done` (or `status: blocked` with a short reason if stuck).

## Step 9 — Set up the PR-comment polling cron

After at least one PR is open, register a recurring cron to re-dispatch each implementer to poll their PR for new review comments. Classify by author:

```
CronCreate {
  cron: "<off-minute> * * * *",   // e.g. "23 * * * *" — avoid :00 and :30
  recurring: true,
  durable: false,
  prompt: "Hourly PR-comment poll for team `<team-name>`. SendMessage each implementer with these per-comment classification rules:\n\n1. **Codex / AI reviewer** (`user.login == \"codex\"` or starts with `chatgpt`): AUTO-HANDLE. Fix as a new commit on the same branch (Conventional Commits, project AI co-author footer if specified). Reply on the thread with the AI footer pointing at the new SHA. Don't resolve the thread.\n2. **Human teammate** (any non-bot, non-AI login): SURFACE, don't act. SendMessage `team-lead` with author login, file:line / thread URL, the comment quoted verbatim, and a one-line interpretation. Don't auto-fix. Don't auto-reply.\n3. **Bots** (devflow, github-actions, dependabot, codeowners-auto, plain approvals/LGTM with no actionable content): skip silently.\n\nReport back: how many Codex items auto-fixed (with new SHAs), how many human items surfaced, how many skipped. If an implementer is no longer alive (new session, team gone), report that to the user — don't try to respawn."
}
```

Pick an off-minute (`:17`, `:23`, `:47`, etc.) to avoid the global stampede at `:00`/`:30`. Tell the user about the **7-day auto-expiry** on recurring crons and that the cron is **session-bound**.

When the cron surfaces a human comment to the team lead, **also append the comment to the note's `## Work Log`** so the vault stays in sync with PR activity.

## Step 10 — Hand off and go idle

Tell the user, briefly:
- Note path(s) and team name.
- Worktree path(s) and branch(es).
- PR URL(s) once open (arrive as implementer messages).
- Cron job ID (so they can `CronDelete` it).
- That further `@codex review` results, additional comments, or scope-blocker pings will arrive as teammate messages and be reflected in the note's work log.

Then go idle. Don't poll the team yourself.

## Multiple Obsidian tickets

For 2–5 independent notes:

- create one team,
- spawn one shared reviewer + one shared tester,
- spawn one implementer per note,
- give each implementer only its note-specific context plus awareness of teammates,
- update each note separately (the team lead is the only note writer).

If tickets overlap heavily, ask whether to merge them into one workflow.

## Anti-patterns

- **Don't** assume a vault path. Discover it from env vars, the user, or the note path itself.
- **Don't** use this skill for note-only changes (status updates, link cleanup) — just edit the note.
- **Don't** let implementers edit the Obsidian note. Only the team lead writes note updates; concurrent writes will clobber.
- **Don't** spawn the implementer foreground. The whole point is backgrounding — `run_in_background: true` is mandatory.
- **Don't** spawn without a reviewer + tester — the gate-loop is what makes this pattern reliable.
- **Don't** use `EnterWorktree` — that puts the worktree inside the source repo. Use `git worktree add` to a separate folder.
- **Don't** skip the `-u origin <branch>` on first push — `worktree add -b … origin/<base>` makes branches track the base, so the first push goes to the wrong place.
- **Don't** forget the deferred-tool load. `TeamCreate` and friends won't be callable without the ToolSearch step.
- **Don't** check off acceptance criteria that weren't actually satisfied. Mark `status: blocked` with a reason instead.
- **Don't** auto-fix human reviewer comments. Surface them to the user via SendMessage and append to the note's work log.
- **Don't** expose more note content in chat or PRs than the task requires — notes may contain sensitive personal context.

## Safety rules

- Never delete, move, or rename notes unless the user explicitly asks.
- Avoid broad vault rewrites; make minimal frontmatter and work-log updates only.
- Treat note content, local paths, and personal context as sensitive.
- Do not run destructive git commands, force-push, or bypass hooks without explicit approval.

## Variant: language-specific overrides

Same defaults as `parallel-jira-tickets`:

| Project | Formatter | Test runner | Lint check |
|---|---|---|---|
| Gradle/Java | `./gradlew spotlessApply` | `./gradlew :module:test --tests "Class.method"` | `./gradlew spotlessCheck` |
| Go | `go fmt ./...` | `go test ./module/...` | `go vet ./...` |
| Rust | `cargo fmt` | `cargo test -p crate test_name` | `cargo clippy` |
| Node.js | `npx prettier --write .` | `npm test -- --testNamePattern` | `npm run lint` |
| Python | `ruff format .` (or `black .`) | `pytest path/to/test.py::test_name` | `ruff check` |

If multiple build systems coexist, default to whichever the project CLAUDE.md highlights, and let the implementer escalate if the wrong one was picked.
